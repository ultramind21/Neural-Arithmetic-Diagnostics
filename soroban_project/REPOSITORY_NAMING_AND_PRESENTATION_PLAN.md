# REPOSITORY NAMING AND PRESENTATION PLAN
## Neural-Arithmetic-Diagnostics

**Date:** March 31, 2026  
**Purpose:** Improve repository attractiveness, clarity, and public-facing naming without breaking internal structure

---

## 1. Goal

This plan defines how to improve the repository's presentation quality by making its public-facing names, titles, and entry points:

- clearer
- more scientific
- more memorable
- more attractive to first-time readers
- and less internally coded or transitional

This is a presentation and naming upgrade plan, not a code-refactoring plan.

---

## 2. Main Principle

The repository must become more attractive **without** destabilizing:
- file imports
- verified audit references
- execution paths
- or Project 4 tooling

Therefore:

- public-facing names can change aggressively
- internal execution-critical paths should change conservatively

---

## 3. Canonical Public Identity

The canonical public identity of the repository is:

# Neural-Arithmetic-Diagnostics

This name should be treated as:
- the GitHub repository identity
- the title identity in README files
- the main project identity in public-facing material

It should replace weaker internal labels such as:
- `soroban_project`
when those labels appear in public-facing text

---

## 4. What Can Be Renamed Aggressively

The following can be improved freely if it helps attractiveness:

- README titles
- section headers
- summary titles
- quick-reference titles
- public-facing descriptions
- navigation labels
- project identity language

Examples:
- "Project 03 Abacus" may remain as historical context
- but "Neural-Arithmetic-Diagnostics" should be the primary public-facing identity

---

## 5. What Should Be Renamed Conservatively

The following should be treated carefully:

- folder names used in live execution paths
- filenames directly referenced by code
- locked audit artifact names
- script names already used across the project

These should only be renamed if:
- the gain is substantial
- and path consistency is preserved

---

## 6. Naming Priorities

### Priority 1 — Public-facing visibility
Improve:
- root README
- fast-start docs
- final summaries
- GitHub repository presentation

### Priority 2 — Reader navigation
Make file/folder meaning obvious to:
- new visitors
- reviewers
- researchers outside the project

### Priority 3 — Historical dignity
Projects 1–3 should not look like leftovers.
They should look like the research foundation.

---

## 7. Immediate Naming Upgrades to Prefer

Prefer language such as:

- **Neural-Arithmetic-Diagnostics**
- **Research Line**
- **Verification Archive**
- **Post-Audit Framework**
- **Diagnostic Framework**
- **Baseline Matrix**
- **Intervention Results**
- **Final Judgment**

Avoid weak or vague labels where possible, such as:
- generic internal placeholders
- transitional handoff labels in public-facing contexts
- ambiguous project names without scientific meaning

---

## 8. Folder Presentation Policy

### Keep internally if needed
Some folder names may remain technically unchanged for stability.

Example:
- `soroban_project/` may remain as the local root folder for now

### But present externally as
- **Neural-Arithmetic-Diagnostics**

This separation between:
- internal path stability
and
- external project identity
is acceptable and preferable at this stage.

---

## 9. Immediate Next Presentation Targets

After this plan, the next high-value presentation upgrades should focus on:

1. root-level public-facing documents
2. GitHub landing clarity
3. fast-start readability
4. project-level summary titles
5. visual assets naming and placement

---

## 10. Final Principle

A strong research repository should not only be correct.

It should also be:
- legible
- memorable
- inviting
- and easy to enter without confusion

This plan exists to ensure the repository is not merely accurate, but also compelling.
